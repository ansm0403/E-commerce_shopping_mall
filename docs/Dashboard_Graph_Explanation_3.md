### 1.1 (계획상) 화면에서 보이는 모습

여러 개의 **사각형 박스**가 한 직사각형 영역 안에 빈틈없이 채워져 있는 모양:

```
┌──────────────────┬──────────────────┐
│                  │   가전 ₩2.4M     │
│   의류 ₩5.6M     │   ▲ 24.1%       │
│   ▲ 12.3%        ├────────────┬─────┤
│                  │ 도서 ₩1.1M │ 잡화 │
│                  │  ▼ 5.0%   │ ... │
└──────────────────┴────────────┴─────┘
```

- **박스 크기 = 매출액** (큰 카테고리일수록 박스가 큼)
- **박스 색 = 전월 대비 증감률**
  - 진한 초록: > +10% (잘 자람)
  - 연한 초록: 0 ~ +10%
  - 노랑: -10 ~ 0%
  - 빨강: < -10% (역성장)
- **박스 라벨**: 카테고리명 + 매출액 + 증감 화살표

### 1.2 7개 항목 정리

#### 1) 이 그래프의 기본 종류

**트리맵 (Treemap)**. ECharts `series: { type: 'treemap' }`.

- "트리(tree)" 구조의 데이터를 직사각형 분할로 표현.
- 박스 크기로 한 변수(매출)를 표현하고, **색**으로 또 하나의 변수(증감률)를 표현.

> 용어:
> - **트리(tree) 구조**: 카테고리 → 하위 카테고리 → 또 하위 카테고리처럼 계층 있는 데이터. 트리맵은 그 계층을 박스 안의 박스(nesting)로 표현 가능.
> - **2차원 인코딩**: 한 도형이 두 가지 정보(크기 + 색)를 동시에 보여주는 방식. 트리맵의 강점.

#### 2) 어떤 데이터를 표현하는가 (계획)

**기간 내 카테고리별 매출 합계 + 전기 대비 증감률**.

```ts
interface CategoryRevenueResponse {
  current:  Array<{ categoryId, name, revenue }>;       // 이번 기간
  previous: Array<{ categoryId, name, revenue }>;       // 전기
  diff:     Array<{ categoryId, name, revenue, deltaPercent }>;  // 박스 그릴 때 사용
}
```

데이터 출처: `orders` + `order_items` + `products` + `categories` JOIN, 결제 완료(`paid_at IS NOT NULL`)된 주문만 합산.

#### 3) 왜 이 데이터에 트리맵인가

- **카테고리가 6개 이상** — 우리 프로젝트의 시드 데이터엔 카테고리 트리가 있고 leaf까지 더하면 더 많음.
- 카테고리 6개 이상에서 **파이/도넛 차트는 가독성이 무너짐** (조각이 가늘어지고 라벨 겹침).
- 트리맵은 박스 크기 + 색의 **2차원 인코딩**으로 "어디가 크고 어디가 잘 자라는가" 두 정보를 동시 표현.
- **멀티 셀러 오픈마켓**의 핵심 지표 — 셀러 모집 전략, 카테고리 트렌드 변화 감지, 수수료 정책 근거. 일반 단일 셀러 쇼핑몰엔 없는 차별화 지표 (Design_Dashboard_log.md Part 4).

#### 4) 장점

- **두 정보를 한 도형에 동시 표현** — 크기(매출) + 색(증감). 막대로는 두 정보를 그리려면 그래프 2개 필요.
- **공간 효율** — 박스가 빈틈 없이 채워짐 → 한정된 카드 면적에 최대한 많은 카테고리 표시.
- **클릭 드릴다운 자연스러움** — 박스 클릭 → 해당 카테고리 주문 목록 페이지로 이동 (`/admin/orders?categoryId=...&startDate=...`).
- **색만 봐도 위험 신호 인지** — 빨강 박스가 크면 "큰 카테고리가 역성장 중" 즉시 인지.

#### 5) 같은 데이터에 다른 그래프를 쓰면 생기는 단점

- **파이/도넛 차트**: 카테고리 6개 이상이면 조각이 가늘어지고 라벨 겹침. 또 "증감률" 정보를 그릴 자리가 없음.
- **막대 그래프**: "크기 비교"는 가능하지만 "증감률"을 같이 그리려면 시리즈 2개로 분리 (높이=매출, 추가 막대=증감) → 시각적 부담.
- **스택 막대**: 1개 막대에 모든 카테고리를 쌓으면 비율 비교는 되지만 절대값(매출액) 비교가 어려움.
- **테이블**: 정확한 숫자는 보이지만 "직관적 비교"가 약함. 위험 신호가 안 튐.

#### 6) 구현 난이도

**어려움**. (아직 미구현이지만) 예상되는 함정:

1. **카테고리 구조 처리** — 우리 프로젝트는 카테고리에 계층(루트/리프)이 있음. v1 설계는 **leaf 카테고리 그대로** 그리고, 추후 루트 매핑은 v2로 미룸 (Design_Dashboard.md §7.2 Chart 4).
2. **카테고리 스냅샷 부재** — `order_items`에 주문 시점 카테고리가 저장돼 있지 않음 → `products.category_id`로 JOIN. 주문 후 상품 카테고리가 바뀌면 과거 매출이 새 카테고리로 잡힘. 포트폴리오 수준에선 허용 (Design_Dashboard.md §12).
3. **색상 함수** — `deltaPercent` 값에 따라 4구간으로 색 결정 (진초/연초/노랑/빨강). ECharts `itemStyle.color`를 함수로 처리. **null guard 필수** (Phase 3 보안 차트와 동일 패턴).
4. **drill-down 라우팅** — `onEvents={{ click: handleClick }}` → 카테고리 ID 추출 → `router.push`.
5. **라벨 가독성** — 작은 박스에 글자가 안 들어가면 자동 숨김 처리 필요 (`label.show` 조건).

#### 7) 구현할 때 주의해야 할 점 (계획 기준)

| 영역 | 주의점 |
|------|--------|
| **카테고리 JOIN** | `products.category_id`로 JOIN (스냅샷 없음) — 한계로 인지 |
| **루트 vs leaf** | v1은 leaf 그대로. v2에서 루트로 ROLLUP 검토 |
| **색상 함수** | deltaPercent → 4구간 분기. null guard 잊지 말 것 |
| **빈 데이터** | 결제 안 된 카테고리는 박스 없음 — 사용자가 "왜 안 보이지?" 혼동할 수 있음. 적절한 안내 필요 |
| **클릭 드릴다운** | `/admin/orders?categoryId=...&startDate=...` 형식 통일 |
| **`paid_at IS NOT NULL`** | 결제 완료된 주문만 매출에 합산. 취소는 미포함 |
| **데이터 크기** | 박스 너무 작으면 라벨 숨김 (사용자가 hover로만 볼 수 있게) |

## 2. 종합 정리 — 다른 사람에게 한 번에 설명하기 위한 핵심 요약

### 2.1 한눈에 보는 비교 표

| 그래프 | 종류 | 핵심 메시지 | 데이터 소스 | 핵심 시각 트릭 | 난이도 |
|--------|------|-----------|-----------|--------------|--------|
| KPI 카드 | 숫자 카드 | "오늘 숫자" | orders, audit_logs (오늘분) | delta% + 색상 (▲초록/▼빨강) | 쉬움 |
| ① 꺾은선 | Line + 점선 비교 | "시간이 지나면서 어떻게 변했나" | audit_logs (action별 일별 카운트) | 전기 점선 오버레이 | 보통 |
| ② 보안 | Bar + Line + 이중 Y축 + markLine | "건수 + 실패율 + 임계초과 위험" 한 화면 | audit_logs (LOGIN/FAILED/LOCKED) | 임계 초과 색상 분기, markLine 10% | 어려움 |
| ③ 펀넬 | Funnel | "단계별 이탈을 깔때기로" | orders (paidAt/shippedAt 등 NULL 제외 COUNT) | sort:'none' + value=rate(%) | 어려움 |
| ④ 트리맵 (미구현) | Treemap | "어디가 크고 어디가 잘 자라나" | orders + order_items + products + categories | 박스 크기=매출, 색=증감률 (2차원 인코딩) | 어려움 |

### 2.2 "왜 이 그래프를 골랐는가" — 한 문장씩

1. **KPI 카드**: 운영자가 페이지를 켜자마자 "오늘 정상인가"를 0.5초 안에 파악해야 하기 때문에, 그래프보다 큰 숫자가 더 빠름.
2. **꺾은선**: "시간이 지나면서 변하는 추세"는 선의 기울기로 한눈에 읽힘. 점선으로 전기와 직접 비교 가능.
3. **보안 복합 차트**: 단위가 다른 "건수(0~30)"와 "비율(0~100%)"을 한 화면에서 같이 봐야 했고, 임계선(10%) 초과 시 시각적 알람이 필요했음. 이중 Y축 + 막대 + 선 + markLine 조합이 유일한 답.
4. **펀넬**: 우리는 주문 ≠ 결제 분리 시스템이라 "어느 단계에서 사용자가 빠지는가"가 매출 직결. 깔때기 모양이 이탈을 시각적으로 강제.
5. **트리맵 (예정)**: 멀티 셀러 오픈마켓 차별화 지표. 카테고리 6개 이상 + 매출 + 증감률을 한 화면에 표현하려면 트리맵 외엔 선택지가 없음.

### 2.3 모든 그래프에 공통 적용된 함정 (반복해서 만난 것들)

| 함정 | 어디서 만나는가 | 해결 방법 |
|------|--------------|----------|
| KST/UTC 시간 어긋남 | 모든 일별 GROUP BY | SQL에 `AT TIME ZONE 'UTC' AT TIME ZONE 'Asia/Seoul'` 이중 캐스트 |
| 빈 날짜 누락 → 선 끊김 | 꺾은선, 보안 차트 | 백엔드 `fillEmptyDates` 헬퍼로 0 채움 |
| ECharts SSR 불가 | 모든 ECharts 차트 | `dynamic(() => import('echarts-for-react'), { ssr: false })` |
| 옵션 교체 중 이전 dataIndex로 콜백 호출 | 보안, 펀넬, (예정)트리맵의 itemStyle.color / tooltip.formatter | 콜백 첫 줄에 `if (!p) return ...` null guard |
| 시리즈 잔존 | compare 토글 등 시리즈 개수 변경 시 | `notMerge: true` |
| numeric/COUNT 결과가 문자열 | 모든 SQL aggregate | `::text` 캐스팅 + 서비스에서 `Number()` |
| camelCase 컬럼 따옴표 | `"createdAt"`, `"paidAt"` 등 | SQL 안에서 쌍따옴표 필수. 안 그러면 소문자화돼 컬럼 못 찾음 |

### 2.4 발표/면접용 한 줄 요약 (다른 사람에게 설명할 때)

> "관리자 대시보드에 **그래프 4종 + KPI 카드 4종**을 만들었습니다.
> ① **일별 주문/결제 추세는 꺾은선** — 추세 + 전기 점선 비교.
> ② **보안 차트는 막대+선 이중 Y축** — 건수와 실패율을 한 화면에 같이 보고, 10% 임계선 markLine으로 시각 경고.
> ③ **결제 전환은 펀넬** — 주문→결제→배송→완료 단계별 이탈을 깔때기로 직관 표현. PostgreSQL `COUNT(컬럼)` NULL 제외 트릭으로 단일 쿼리 5단계 집계.
> ④ **카테고리 매출은 트리맵 (설계만)** — 박스 크기=매출, 색=증감으로 2차원 인코딩.
> 모든 그래프는 ECharts로 통일했고(펀넬·트리맵까지 한 라이브러리로 그릴 수 있는 유일한 선택), URL 쿼리 파라미터(`?startDate=&endDate=`)를 상태 진실 원천으로 두어 새로고침·공유·뒤로가기에 모두 안전합니다."
