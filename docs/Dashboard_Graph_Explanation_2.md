## 1. 그래프 ② — 로그인 보안 차트 (복합 차트, 이중 Y축)

> 구현:
> - 옵션 빌더: `frontend/src/lib/charts/security.ts`
> - 컴포넌트: `frontend/src/app/(admin)/admin/dashboard/components/SecurityChart.tsx`
> - 데이터 출처: `audit_logs` 테이블 (LOGIN / FAILED_LOGIN / ACCOUNT_LOCKED)
>
> **이 차트는 5종 중 가장 어려운 차트다.** 이유는 후술.

### 1.1 화면에서 보이는 모습

가로축 = 날짜.
세로축이 **양쪽 두 개** 있음:

- **왼쪽 Y축**: 건수 (0, 5, 10, 15…). 막대 그래프가 이 축을 씀.
- **오른쪽 Y축**: 실패율 % (0~100). 꺾은선이 이 축을 씀.

그 위에 3개 시리즈가 함께 그려짐:

- **주황(또는 빨강) 막대** — "로그인 실패" 건수 (FAILED_LOGIN)
- **보라 막대** — "계정 잠금" 건수 (ACCOUNT_LOCKED)
  - 위 둘은 **stack** 으로 위아래로 쌓임 (하루 총 보안 이벤트 막대가 됨)
- **파란 선** — 실패율 % (failed / (success + failed) × 100)
- **빨간 점선 (markLine)** — 10% 경고선. 실패율이 이 선을 넘으면 위험.
- 실패율이 10% 넘는 날은 **막대 색이 주황 → 빨강으로 바뀜**.

### 1.2 7개 항목 정리

#### 1) 이 그래프의 기본 종류

**복합 차트(Combo Chart) — 막대(Bar) + 선(Line) + 이중 Y축 + 스택(Stack) + 경고선(markLine)**.

- 한 그래프 안에 **종류가 다른 시리즈(막대 + 선)** 가 같이 들어감 → "복합".
- Y축이 **두 개** 라서 단위가 다른 값들을 한 화면에 그릴 수 있음 → "**이중 Y축(dual Y-axis)**".
- 두 막대를 위로 쌓아 표시 → "**스택 막대(stacked bar)**".
- 임계선(threshold)을 점선으로 표시 → "**markLine**".

> 용어:
> - **이중 Y축**: 같은 X축(날짜)을 공유하지만, 세로축이 좌·우 두 개. 단위가 다른 값을 비교할 때 씀.
> - **스택 막대**: 막대 두 개를 따로 두지 않고 위로 쌓아 "합계 모양"이 보이게 만든 막대.
> - **임계선 (markLine)**: 차트 위에 가로/세로로 그어주는 기준선. "이 선을 넘으면 위험"이라는 기준값을 시각적으로 표시.

#### 2) 어떤 데이터를 표현하는가

**보안 관련 audit 로그 일별 집계**:

| 행동(action) | 의미 | 차트에서 |
|------------|------|---------|
| `LOGIN` | 로그인 성공 | 화면엔 직접 표시 X (실패율 계산 분모로만 사용) |
| `FAILED_LOGIN` | 로그인 실패 | 주황/빨강 막대 (왼쪽 Y축) |
| `ACCOUNT_LOCKED` | 계정 잠금 | 보라 막대 (왼쪽 Y축, stack) |
| (파생) `실패율 = failed / (success + failed) × 100` | 시도 대비 실패 비율 | 파란 선 (오른쪽 Y축, 0~100%) |

#### 3) 왜 이 데이터에 복합 차트인가

핵심 이유 = **단위와 스케일이 완전히 다른 두 값을 한 화면에 같이 봐야 하기 때문**.

| 값 | 단위 | 일반적 범위 |
|----|------|----------|
| 로그인 성공 (LOGIN) | 건 | 300~500 건/일 |
| 로그인 실패 (FAILED_LOGIN) | 건 | 10~30 건/일 |
| 계정 잠금 (ACCOUNT_LOCKED) | 건 | 0~5 건/일 |
| 실패율 | % | 0~100 |

→ 같은 Y축에 다 그리면:
- 성공(400)이 너무 커서 실패(20)나 잠금(2)은 막대가 **납작해서 안 보임**.
- "오늘 실패율 12% (위험)"인지가 막대 길이만 봐선 알 수 없음.

**해결**: Y축을 두 개로 분리.
- 왼쪽 = 건수 (작은 값들이 잘 보이도록)
- 오른쪽 = 실패율(%) (0~100 고정)

그리고 "실패율 10% 넘으면 경고"를 표현하기 위해 **markLine 10%** 를 추가.

또 **"실패율이 임계 초과한 날은 막대를 빨강으로"** 같은 **데이터 기반 색상 분기**까지 추가 → 운영자가 그래프를 그냥 흘려봐도 위험한 날이 시각적으로 튐.

#### 4) 장점

- **단위가 다른 두 지표(건수 + %)를 동시에** 한눈에. 운영자가 "건수도 많고 비율도 높네" 같은 복합 판단 가능.
- **stacked bar** — 보안 이벤트 총량을 막대 높이로, 종류 비율을 색으로 동시 표현.
- **markLine + 색상 분기** — 위험한 날을 안 봐도 보임. "그래프를 응시하지 않아도 알람을 받음" 효과 (1차 설계 논의에서 강조된 운영 대시보드 UX 원칙 — "운영자는 24시간 응시 못함").

#### 5) 같은 데이터에 다른 그래프를 쓰면 생기는 단점

- **막대만 (3개 시리즈 grouped bar)**: 실패율(%)을 막대로 표현하면 0~100 범위 + 건수(0~30 범위) 막대와 같은 Y축에 놓이게 됨 → 건수 막대가 시각적으로 짓눌림. 또 % 값은 막대보다 선이 추세 읽기 쉬움.
- **선만 (3개 시리즈 line)**: "건수"는 막대로 그려야 직관적(특히 이산값). 선은 연속적인 흐름에 맞고 막대는 일별 카운트에 맞음.
- **이중 Y축 없이 한 축**: 스케일 차이로 작은 값이 사라짐 (3번에서 설명).
- **임계선 없이**: "10%가 위험"이라는 약속을 차트가 알려주지 못함 → 운영자가 직접 계산해야 함.

#### 6) 구현 난이도

**어려움**. 이번 대시보드 5종 중 가장 까다로움.

어려운 부분:

1. **이중 Y축 설정** — `yAxis`를 배열로 두 개 정의하고, 각 series에 `yAxisIndex: 0` 또는 `1`을 명시해야 함. 빼먹으면 모든 시리즈가 왼쪽 축으로 몰림.
2. **스택 막대 설정** — 두 막대 series에 같은 `stack` 이름(`stack: 'security'`)을 부여해야 위로 쌓임.
3. **markLine 위치** — `markLine`은 시리즈 안에 정의해야 하며, 어느 Y축을 기준으로 할지 정해짐 (실패율 선 시리즈 안에 두면 오른쪽 Y축 10에 그어짐).
4. **`itemStyle.color` 콜백** — "임계 초과 시 빨강"을 위해 각 막대 색을 데이터에 따라 다르게 → 함수로 동적 결정. 처음 보는 패턴.
5. **`tooltip.formatter` 콜백** — 한 박스에 5줄 정보를 어떻게 정리할지 HTML 문자열 직접 작성.
6. **❗ null guard 함정** — ECharts는 옵션을 교체하는 잠시 동안 **이전 dataIndex로 콜백을 호출**할 수 있음. 데이터 길이가 줄어들었을 때 `data.daily[i]`가 `undefined`라서 throw됨. 그래서 `if (!p) return ''` / `if (!p) return COLOR_BAR_NORMAL` 같은 방어 코드 **필수**. (실제 Phase 3에서 발생한 버그 — Handoff_3.md §1.3)

#### 7) 구현할 때 주의해야 할 점

| 영역 | 주의점 |
|------|--------|
| **Y축 인덱스** | `series[].yAxisIndex` 명시 — 0=왼쪽(건수), 1=오른쪽(실패율). 빼먹으면 한쪽에 몰림 |
| **stack 이름** | 두 막대 series에 동일한 `stack: 'security'` 부여. 한쪽만 있으면 안 쌓임 |
| **오른쪽 Y축 고정 범위** | `min: 0, max: 100` 명시 — 실패율은 %라 범위가 정해져있음. 자동 스케일링되면 비교 어려움 |
| **markLine은 line 시리즈 안에** | 막대 시리즈 안에 두면 왼쪽 Y축 10에 그어져 의미 없음 |
| **임계 색상 콜백 null guard** | `if (!p) return COLOR_BAR_NORMAL` — 옵션 교체 중 dataIndex 불일치 방어 |
| **툴팁 fallback** | `if (!p) return ''` — 빈 데이터일 때 ECharts가 throw 못하도록 |
| **`failureRate` 계산 위치** | DB 쿼리에서 안 하고 백엔드 서비스 레이어에서 계산. 이유: 빈 날짜(0/0)일 때 NaN 안 나오게 `total > 0` 분기 |
| **`fillEmptyDates` 순서** | SQL → Number 캐스팅 → fillEmptyDates → failureRate 계산 → DTO 빌드. 순서 어기면 NaN |
| **X축 라벨 회전** | 14일 이상이면 `rotate: 30` — 안 그러면 라벨이 겹침 |
| **색상 의미** | 빨강=위험, 주황=경고, 보라=참고. 색 의미가 흔들리면 운영자가 헷갈림 |

## 2. 그래프 ③ — 결제 전환 펀넬 (Funnel Chart)

> 구현:
> - 옵션 빌더: `frontend/src/lib/charts/funnel.ts`
> - 컴포넌트: `frontend/src/app/(admin)/admin/dashboard/components/FunnelChart.tsx`
> - 데이터 출처: `orders` 테이블 (createdAt, paidAt, shippedAt, deliveredAt, completedAt 컬럼)

### 2.1 화면에서 보이는 모습

위가 넓고 아래로 갈수록 좁아지는 **사다리꼴 5층 케이크** 같은 모양:

```
┌────────────────────────────────┐  주문 생성 — 412건 (100%)
└──┐                          ┌──┘
   │  결제 완료 — 308건 (75%)   │
   └──┐                    ┌──┘
      │ 배송 중 — 246건 (60%) │
      └──┐              ┌──┘
         │ 배송완료 — 215건 (52%) │
         └──┐        ┌──┘
            │ 구매확정 — 178건 (43%) │
            └────────┘
```

각 단계의 **너비 = 전체(1단계) 대비 도달률(%)**.
색은 위에서 아래로 진해지는 단조 그라디언트 (연한 하늘 → 진한 파랑).

### 2.2 7개 항목 정리

#### 1) 이 그래프의 기본 종류

**펀넬 차트 (Funnel Chart)** — "**깔때기**" 모양 그래프.
- 사용자가 어떤 흐름(주문 → 결제 → 배송 → ...)을 거치면서 **각 단계에서 몇 %가 다음으로 넘어가는지** 표현.
- ECharts에서는 `type: 'funnel'` 시리즈 하나로 그림.

> 용어:
> - **전환률(conversion rate)**: 한 단계에서 다음 단계로 넘어간 비율. 예: 주문 100건 중 결제 75건 → 전환률 75%.
> - **이탈률(drop rate)**: 직전 단계에서 떨어져 나간 비율. 위 예시면 25%.
> - **코호트(cohort)**: "같은 조건으로 묶은 사용자 집합". 여기서는 "기간 안에 **주문 생성된** 주문 그룹"을 코호트로 묶음. 이 그룹이 시간이 지나 몇 %나 끝까지 도달했는지를 봄.

#### 2) 어떤 데이터를 표현하는가

`orders` 테이블 한 번의 쿼리로 5단계 카운트.

| 단계 | DB에서 어떻게 카운트? | 의미 |
|------|------------------|------|
| 1. 주문 생성 (created) | `COUNT(*)` | 기간 내 만들어진 모든 주문 |
| 2. 결제 완료 (paid) | `COUNT("paidAt")` | paidAt이 NULL이 아닌 주문 |
| 3. 배송 중 (shipped) | `COUNT("shippedAt")` | shippedAt이 NULL이 아닌 주문 |
| 4. 배송 완료 (delivered) | `COUNT("deliveredAt")` | deliveredAt이 NULL이 아닌 주문 |
| 5. 구매 확정 (completed) | `COUNT("completedAt")` | completedAt이 NULL이 아닌 주문 |

> 🧠 **PostgreSQL 트릭**: `COUNT(컬럼명)`은 **NULL을 제외한 카운트**. 그래서 `paidAt`이 비어있는 주문은 `COUNT("paidAt")`에 안 잡힘. 이 한 줄 트릭 덕에 한 번 쿼리로 5단계 모두 집계 가능.

각 단계에서 백엔드가 두 가지를 더 계산:
- **rate**: 1단계(created) 대비 도달률. 예: paid 308 / created 412 = 75%
- **dropRate**: 직전 단계 대비 이탈률. 예: paid 308 → shipped 246, 직전 대비 (246/308) = 80% 도달 → 20% 이탈

#### 3) 왜 이 데이터에 펀넬인가

- **단계별로 줄어드는 흐름** — "위가 넓고 아래로 갈수록 좁아진다"는 시각적 메타포가 이탈을 직관적으로 표현. 폭이 그대로 도달률.
- 우리 프로젝트는 **주문 ≠ 결제** 가 분리된 시스템. 그래서 "주문은 만들어졌는데 결제 단계에서 빠진 비율"이 중요 매출 지표.
- 4단계 이상 단계가 있어 막대 5개로 그리면 "단계 순서가 머릿속에서 정렬되지 않음". 펀넬은 위→아래 흐름이 강제됨.

> **왜 단순 카운트가 아니라 코호트인가?**
> 단순 카운트: "기간 내 PAYMENT_VERIFIED 이벤트 수 / 기간 내 ORDER_CREATED 이벤트 수" — 잘못된 답. 4/27에 결제된 주문이 4/15에 만들어졌으면 그 둘이 같은 주문인지 무관하게 그냥 더해버림.
> **코호트 방식**: "기간 안에 생성된 주문만" 코호트로 묶고, **그 주문들이** paid/shipped/... 단계에 얼마나 도달했는지를 봄. 같은 주문 그룹의 추적이라 의미가 정확함.

#### 4) 장점

- **이탈을 시각적으로 즉시 인지** — 어느 단계에서 폭이 갑자기 좁아지면 그게 병목.
- **단계별 라벨에 건수 + %를 동시 표시** — 절대값(412건)과 상대값(100%) 둘 다 확인.
- 단일 쿼리(orders 테이블 한 번)로 5단계 모두 집계 가능 → 백엔드 성능 매우 좋음.
- 결제 게이트웨이 장애 같은 운영 이슈를 즉시 감지 가능 (paid 단계 폭이 갑자기 줄어들면 신호).

#### 5) 같은 데이터에 다른 그래프를 쓰면 생기는 단점

- **막대 그래프 (세로 5개)**: 단계 순서가 차트 자체로 강제되지 않음 (정렬 안 하면 키 순서로 나옴). "이탈"이 막대 길이 차로만 보여서 직관성 떨어짐.
- **파이/도넛 차트**: 단계 간 흐름이 표현 안 됨. "전체 중 비율"만 보임 → 부적합.
- **꺾은선**: 단계는 연속적인 수치 변화가 아니라 이산 단계(주문→결제→배송) → 선으로 잇는 게 어색.
- **테이블**: 숫자는 정확하지만 "어느 단계가 크게 떨어졌는지"가 시각적으로 안 와닿음.

#### 6) 구현 난이도

**어려움**. 보안 차트보다는 살짝 쉽지만 여전히 함정이 있음.

어려운 부분:

1. **`sort: 'none'` 빼먹기 함정** — ECharts funnel은 기본적으로 **값이 큰 단계부터 자동 정렬**함. 그러면 단계 순서가 망가짐 (1단계 created가 가장 크니까 다행히 위에 가지만, dropRate 계산이 완전히 어긋남). 반드시 `sort: 'none'` 명시.
2. **`value`에 무엇을 넣을지** — 단계의 너비를 결정하는 값. 현재 구현은 **rate(%)** 사용 → 시각적 너비 = 도달률. count를 넣으면 큰 값이 너무 압도적이라 작은 단계가 거의 안 보임.
3. **`minSize` 설정** — 마지막 단계가 1%처럼 작으면 화면에서 사라짐 → `minSize: '15%'` 같이 최소 폭 보장.
4. **dropRate 계산** — 1단계는 dropRate 0(직전이 없음). 분기 처리 필요.
5. **❗ null guard** — 보안 차트와 동일 패턴. `tooltip.formatter`/`label.formatter` 콜백에서 `if (!s) return ''`.

#### 7) 구현할 때 주의해야 할 점

| 영역 | 주의점 |
|------|--------|
| **`sort: 'none'`** | **빼먹지 말 것.** 빠지면 단계 자동 정렬됨 → 이탈률 의미가 깨짐 |
| **값(value) 선택** | rate(%) 사용 → 도달률을 시각화. count 자체는 label/tooltip에 표시 |
| **`minSize`** | 작은 단계가 사라지지 않게 `'15%'` 정도 보장 |
| **색상 그라디언트** | 단조롭게 진해지는 단방향 (연한 → 진한) — 단계 순서를 색으로도 강조 |
| **`label.formatter`** | `"${name}\n${count}건 (${rate}%)"` — 한 라벨에 정보 3개 |
| **tooltip null guard** | `if (!s) return ''` — Phase 3에서 확립된 ECharts 옵션 교체 보호 패턴 |
| **코호트 정의 명시** | "createdAt이 기간 안에 든 주문에 한정". 기간 안에 결제된 다른 코호트는 제외됨 (예: 4/1 생성 → 4/5 결제도 4/1 코호트). 이 정의가 흐려지면 분석이 무의미 |
| **취소(cancelled)는 펀넬 밖** | 취소는 별도 카운트로 표시. 펀넬에 같이 넣으면 단계 순서 깨짐 |
| **SQL 한 줄 트릭** | `COUNT(paid_at)` = NULL 제외 카운트. 모르면 단계마다 따로 쿼리하느라 5번 호출 |

## 3. 본 문서 요약

- **그래프 ② 보안 차트** — 가장 어려운 차트.
  - 단위가 다른 "건수"와 "비율(%)"을 같이 보여줘야 해서 **이중 Y축 + 막대 + 선 + 스택 + markLine** 5중 복합.
  - 핵심 함정: **null guard 필수** (옵션 교체 중 콜백 보호), **markLine은 line 시리즈 안에**, **fillEmptyDates 순서**.
- **그래프 ③ 펀넬** — 단계별 이탈을 시각적으로 강제 표현하는 모양.
  - PostgreSQL `COUNT(컬럼명) = NULL 제외 카운트` 트릭으로 단일 쿼리 5단계 집계.
  - 핵심 함정: **`sort: 'none'`** (안 넣으면 자동 정렬되어 단계 깨짐), **value에 rate(%) 사용**, **코호트 정의** 명확히.
