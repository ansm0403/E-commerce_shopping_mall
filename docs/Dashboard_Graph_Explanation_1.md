## 0. 사용된 차트 라이브러리 한 줄 정리

- **ECharts** — 중국 Apache 재단의 차트 라이브러리. 펀넬·트리맵까지 한 라이브러리로 다 그릴 수 있어서 선택됨.
- 다른 후보 (Recharts, Chart.js)는 펀넬·트리맵을 지원하지 않아 두세 개 라이브러리를 섞어 써야 했음. 그래서 ECharts 하나로 통일.

> ⚠️ 용어: **차트(chart) = 그래프(graph)**. 이 문서에선 의미 구분 없이 같이 씀.

## 1. 전체 요약 표

대시보드는 총 **4종 그래프 + 1종 요약 카드 묶음**으로 구성된다.
(트리맵은 설계되어 있으나 **아직 미구현** — Phase 5)

| # | 이름 | 그래프 종류 | 보여주는 것 | 왜 이걸 골랐나 (한 줄) | 난이도 | 구현 상태 |
|---|------|-----------|-----------|--------------------|--------|---------|
| — | **KPI 카드 4개** | (그래프 아님 — 큰 숫자 카드) | 오늘 주문/매출/미처리 배송/로그인 실패율 | "그래프 보기 전 한눈에 숫자부터" | 쉬움 | ✅ 완료 |
| 1 | **일별 주문/결제** | 꺾은선 (Line) | 시간이 지나면서 주문·결제·취소가 어떻게 변했는지 | 추세를 가장 빠르게 읽음 | 보통 | ✅ 완료 |
| 2 | **로그인 보안 현황** | **복합 차트** (스택 막대 + 선, 이중 Y축) | 일별 로그인 실패/계정 잠금 건수 + 실패율(%) | "건수"와 "비율"을 한 화면에 같이 봐야 함 | **어려움** | ✅ 완료 |
| 3 | **결제 전환 펀넬** | 펀넬 (Funnel) | 주문이 만들어진 후 단계별로 몇 %가 결제·배송·구매확정까지 도달했나 | 단계별 이탈을 직관적으로 표현 | 어려움 | ✅ 완료 |
| 4 | **카테고리별 매출** | 트리맵 (Treemap) | 어느 카테고리가 얼마나 팔리는지 + 전월 대비 증감 | 6개 이상 카테고리에 파이차트보다 가독성 ↑ | 어려움 | ⚠️ **미구현 (Phase 5 예정)** |

### 1.1 그래프별 데이터가 어디서 오는가 (한 줄)

| 그래프 | 데이터 출처 (DB 테이블) |
|--------|---------------------|
| KPI 4개 | `orders` + `audit_logs` (오늘 분만 조회) |
| ① 꺾은선 (주문/결제) | `audit_logs` — 사용자가 한 행동 기록 테이블 |
| ② 보안 차트 | `audit_logs` — 같은 테이블에서 LOGIN / FAILED_LOGIN / ACCOUNT_LOCKED만 |
| ③ 펀넬 | `orders` — 주문 테이블의 paidAt, shippedAt 같은 타임스탬프 컬럼 |
| ④ 트리맵 (미구현) | `orders + order_items + products + categories` JOIN |

```
├─────────────────────────────────────────────────┤
│  ③ 결제 전환 펀넬                                   │
├─────────────────────────────────────────────────┤
│  ④ 카테고리 트리맵  ← 미구현                         │
└─────────────────────────────────────────────────┘
```

### 1.3 모든 그래프에 공통으로 적용된 UX 원칙 (먼저 알면 차트 이해가 쉽다)

| 원칙 | 설명 |
|------|------|
| **기간 필터** | 화면 상단 필터에서 7일/30일/사용자 지정 날짜로 바꾸면 모든 차트가 동시에 다시 그려짐 |
| **URL이 곧 상태** | 필터 값이 주소창 `?startDate=...&endDate=...`에 저장됨 → 새로고침해도, 링크 공유해도 같은 화면 |
| **빈 날짜 0으로 채움** | DB는 "데이터 있는 날짜만" 반환 → 백엔드에서 빠진 날짜를 0으로 채워 보냄 (안 채우면 선이 끊겨 보임) |
| **KST(한국 시간) 명시 처리** | DB는 UTC로 저장돼 있어 그냥 GROUP BY 하면 한국 자정과 9시간 어긋남 → SQL에 `AT TIME ZONE 'Asia/Seoul'` 명시 |
| **차트 SSR 차단** | ECharts는 `window` 객체가 필요해 서버에서 못 그림 → `dynamic({ ssr: false })`로 클라이언트에서만 그림 |
| **`notMerge=true` + null guard** | 옵션 교체 시 이전 인덱스로 콜백이 잠깐 호출되는 ECharts 특성 때문에 빈 데이터 보호 코드 필수 |

## 2. KPI 카드 4개 (그래프 아니지만 먼저 봐야 하는 것)

> 구현: `frontend/src/app/(admin)/admin/dashboard/components/KpiCards.tsx`

KPI는 "**K**ey **P**erformance **I**ndicator" — 한국어로 **핵심 성과 지표**.
그래프가 "추세"를 보여준다면, KPI 카드는 **"지금 이 순간의 숫자"** 한 개만 크게 보여주는 도구.

### 2.1 카드 4개의 의미

| 카드 | 보여주는 숫자 | 비교 기준 |
|------|------------|----------|
| 오늘 주문 | 오늘 KST 자정~지금까지 만들어진 주문 건수 | 어제 같은 시각까지 (▲/▼ %) |
| 오늘 매출 | 오늘 결제 완료된 주문 금액 합 (₩) | 어제 같은 시각까지 (▲/▼ %) |
| 미처리 배송 | 결제는 됐는데 아직 발송 안 된 주문 수 | (비교 없음 — 현재값만) |
| 로그인 실패율 | 오늘 로그인 실패 / (성공 + 실패) × 100 | 임계값 10% — 넘으면 빨강 |

### 2.2 7개 항목 정리

1. **기본 종류**: 그래프가 아닌 **숫자 카드(Stat Card / Metric Card)**.
2. **표현 데이터**: "오늘 하루치 핵심 지표"를 한 줄에 4개 펼침.
3. **왜 이 표현인가**:
   - 추세보다 **"지금 당장 비정상인가"**를 0.5초 안에 보여줘야 함.
   - 그래프는 시선이 한 번 더 머무름 → "한눈에"는 카드가 압도적으로 빠름.
4. **장점**:
   - 가장 중요한 숫자 4개를 모니터 위쪽에 박아둬, 운영자가 페이지를 켜자마자 인지.
   - **delta(전일 대비 %)** 가 함께 있어 "오늘이 이상한지" 판단 가능. `▲ 12.3%`처럼 화살표 + 색상(초록/빨강).
5. **다른 표현으로 바꾸면 단점**:
   - 막대 그래프로 4개를 그리면 → 막대 길이 비교가 "주문 vs 매출"처럼 단위가 달라 무의미.
   - 표(table)로 만들면 → 값 강조가 약해져 운영자가 놓침.
6. **구현 난이도**: **쉬움**. 그냥 `<div>` 4개 + 인라인 스타일. ECharts 안 씀.
7. **주의점**:
   - **delta 색상 약속**: 양수=초록↑ / 음수=빨강↓ — 단, **로그인 실패율은 반대**. (실패가 줄어드는 게 좋은 것)
     코드에서 `deltaStyle(delta, invert=true)` 옵션이 그 처리를 위한 자리. 현재 KPI엔 invert가 안 쓰이지만 확장성을 위해 남겨둠.
   - 임계값 초과 시 숫자 색을 빨강으로 (`failureExceeded` 분기 — KpiCards.tsx:75).
   - `formatNumber` / `formatCurrency` 로 천 단위 콤마 — 매출 `999000`이 `₩999,000`로 나오게.

## 3. 그래프 ① — 일별 주문/결제 꺾은선

> 구현:
> - 옵션 빌더: `frontend/src/lib/charts/order-trend.ts`
> - 컴포넌트: `frontend/src/app/(admin)/admin/dashboard/components/OrderTrendChart.tsx`
> - 데이터 출처: `audit_logs` 테이블 (ORDER_CREATED / PAYMENT_VERIFIED / ORDER_CANCELLED)

### 3.1 이 그래프가 화면에서 보이는 모습 (말로 그리기)

가로축 = 날짜 (예: 2026-04-21 ~ 2026-04-27 7개 점).
세로축 = 건수.
선이 3~5개 겹쳐 있음:

- **파란 실선** — 주문 (ORDER_CREATED)
- **초록 실선** — 결제 완료 (PAYMENT_VERIFIED)
- **빨간 실선** — 취소 (ORDER_CANCELLED)
- **(전기 비교 ON일 때만) 연한 파란 점선** — 전기 주문
- **(전기 비교 ON일 때만) 연한 초록 점선** — 전기 결제

점선은 "지난 기간 같은 길이와 비교한 모습".
예: 이번 7일이 `4/21 ~ 4/27`이면 전기는 `4/14 ~ 4/20`.

### 3.2 7개 항목 정리

#### 1) 이 그래프의 기본 종류

**꺾은선 그래프 (Line Chart)**.
- 각 날짜에 점을 찍고, 점과 점을 선으로 이은 그래프.
- 실제 구현은 `smooth: true`로 **약한 곡선(spline)** 처리 — 직선보다 추세선이 부드럽게 보임.

> 용어:
> - **시리즈(series)**: 그래프 위에 그려지는 "선 한 개"를 시리즈라고 부름. 이 차트는 3~5개 시리즈가 같이 그려진다.
> - **카테고리 축(category axis)**: 가로축이 "숫자"가 아니고 "날짜 라벨"처럼 정해진 항목 목록인 경우. 여기서 X축이 카테고리 축.

#### 2) 어떤 데이터를 표현하는가

`audit_logs` 테이블에서 **하루치 활동 횟수 3종**을 카운트:

| 행동(action) | 의미 | 차트에서 어느 선 |
|------------|------|----------------|
| `ORDER_CREATED` | 사용자가 주문을 만들었음 (결제 전) | 파랑 실선 |
| `PAYMENT_VERIFIED` | 결제가 완료됨 | 초록 실선 |
| `ORDER_CANCELLED` | 주문이 취소됨 | 빨강 실선 |

이걸 KST 기준 일별로 GROUP BY 해서 7~30개 점이 만들어짐.

> ⚠️ "주문 ≠ 결제" — 우리 프로젝트는 주문이 만들어진 뒤 결제가 별도 단계로 일어남.
> 그래서 `ORDER_CREATED` 100건 발생해도 `PAYMENT_VERIFIED`가 80건이라면 20건은 결제 안 한 것.
> 두 선 사이의 간격 = **매출 누수**. 이걸 한눈에 보여주는 게 이 차트의 핵심 가치.

#### 3) 왜 이 데이터에 꺾은선인가

**"시간이 지나면서 숫자가 어떻게 변했나"는 꺾은선이 최강**. 이유:

| 비교 후보 | 같은 데이터에 쓰면 어떻게 되나 |
|---------|-------------------------|
| 막대 그래프 | 추세(올라가는지 내려가는지)가 한눈에 안 들어옴. 막대 3종 × 7일 = 21개 막대가 들어차서 답답. |
| 영역(Area) 그래프 | 여러 선이 겹치면 색이 섞여서 안 보임. (스택 영역으로 쌓으면 "절대 건수" 비교는 가능하지만, 시리즈 간 격차 보기엔 꺾은선이 명료) |
| 산점도 | 점만 찍히면 "다음 점으로 이어지는 방향"이 보이지 않음 |

또, 꺾은선은 **여러 시리즈를 겹쳐 비교하기 쉬움** — 5개 선을 한 화면에 올려도 색만 잘 나누면 읽힘.

#### 4) 장점

- **기울기 = 변화율**. 선이 가파르면 "급증/급감", 평평하면 "안정". 운영자가 추세를 본능적으로 읽음.
- **두 선의 간격 = 의미 있는 갭**. 예: 파랑(주문)과 초록(결제)이 멀어지면 결제 단계 문제 발생 의심.
- **전기 점선과 직접 비교** — "오늘 200건은 좋은 거야 나쁜 거야?"에 즉답 가능.
  - 이걸 위해 백엔드 응답이 `current`와 `previous`로 분리됨 (`OrderTrendResponse`).

#### 5) 같은 데이터에 다른 그래프를 쓰면 생기는 단점

- **막대 그래프**: 3종 × 7일 = 21개 막대. 카테고리별 막대(grouped bar)로 그리면 "각 날짜 안에서 3종 비교"는 쉬움. 하지만 "**날짜 흐름** 비교"는 어려움 — 시선이 위아래로 튐.
- **영역 차트 (stacked area)**: 누계는 보이는데 개별 시리즈가 가려져서 "취소가 늘었나"가 안 보임.
- **파이 차트**: 시간 축 자체가 사라져서 부적합. (파이는 "한 시점의 구성비"에만 쓸 것)

#### 6) 구현 난이도

**보통**.
- ECharts 기본 line type만 쓰면 어렵지 않음.
- 어려운 부분:
  - **빈 날짜 채우기** — 데이터 없는 날에 점을 찍지 않으면 선이 끊김. 백엔드에서 `fillEmptyDates` 헬퍼로 0을 채워서 보냄 (`time-range.helper.ts`).
  - **전기 비교 로직** — `compareWithPrevious=true`면 같은 길이의 **이전 기간**을 한 번 더 쿼리해서 점선으로 추가. 시리즈 개수가 3 → 5로 바뀌어서 `notMerge=true`로 시리즈 잔존 방지 필요.
  - **KST 시간대 처리** — UTC 그대로 GROUP BY하면 한국 자정과 9시간 어긋남.

#### 7) 구현할 때 주의해야 할 점

| 영역 | 주의점 |
|------|--------|
| 데이터 가공 | `fillEmptyDates`로 빠진 날짜를 `{ ordered:0, paid:0, cancelled:0 }`로 채워야 선이 끊기지 않음 |
| X축 정렬 | 백엔드 SQL에서 `ORDER BY 1 ASC`로 날짜 오름차순 보장. 안 하면 선이 갈지자로 꺾임 |
| 축 단위 | Y축은 정수 (`minInterval: 1`) — 주문 0.5건은 무의미 |
| 색상 | 의미 있는 색 매핑 — 파랑=주문, 초록=결제(긍정), 빨강=취소(부정). KPI 카드와 색 통일 |
| 범례 표시 | `legend.data`에 시리즈 이름 자동 매핑 → 사용자가 클릭해서 특정 선 숨길 수 있음 (ECharts 기본 기능) |
| 툴팁 | `trigger: 'axis'`로 같은 날짜의 5개 시리즈 값을 한 박스에 모음 — `'item'`이면 한 점만 뜸 |
| 반응형 | `<ReactECharts>`는 컨테이너 width에 자동 맞춤. 단, 부모 div가 0이면 안 보이니 명시적 height(360px) 부여 |
| SSR 차단 | `dynamic(() => import('echarts-for-react'), { ssr: false })` — ECharts가 `window`를 참조하기 때문에 서버 렌더 불가 |
| 잔존 시리즈 | compare 토글로 시리즈 개수가 변하면 `notMerge: true` 필수 — 안 그러면 이전 점선이 남음 |

## 4. 본 문서 요약

- 대시보드는 **4종 그래프 + 1종 KPI 카드 묶음**.
- KPI 카드는 "오늘 숫자"만 큼지막하게. 그래프 보기 전 첫 시선.
- **그래프 ①(꺾은선)** 은 "시간에 따른 변화"에 최적. 추세 + 시리즈 비교에 강하고, 같은 차트 위에 전기(점선) 겹쳐서 비교까지.
- 같은 데이터를 막대로 그리면 추세가 안 보이고, 영역으로 쌓으면 시리즈 비교가 흐려짐.
- 난이도 자체는 보통이지만, **KST 시간대 / 빈 날짜 0채움 / SSR 차단 / notMerge** 4가지가 첫 구현자 함정.
